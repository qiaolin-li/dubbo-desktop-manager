# 渲染进程插件设计（现状 + 可扩展方向）

## 一、总体结构与目标

渲染进程插件系统的目标是：在 **Vue2 + ElementUI 的渲染端**，让插件能够在不修改宿主代码的情况下：

- 扩展 UI（菜单、页面、设置面板、对话框、数据源编辑页、服务页/调用页等）
- 使用宿主提供的“能力”（事件总线、剪贴板、通知、配置读写、调用主进程服务等）
- 与主进程插件暴露的服务协作（`plugin.xxx` 的主进程服务）

渲染端插件加载入口在 `src/renderer/core/AppRenderer.js`。

```82:105:C:\Users\PC\.cursor\worktrees\dubbo-desktop-manager\lye\src\renderer\core\AppRenderer.js
async loadPlugin () {
    const pluginInfos = await pluginManagerClient.getPluginModules();
    
    pluginInfos.forEach(async (plugin) => {
        try {
            // 1、创建插件核心对象
            const appPluginCore = new AppRendererPluginCore(this, plugin.pluginName);

            // 2、加载插件国际化文件
            if(plugin.i18nPath) {
                const i18nModule = requireFunc(plugin.i18nPath)(appPluginCore.geti18nRegistrar());
                await i18nModule.install();
            }

            // 3、加载插件渲染逻辑
            if(plugin.rendererPath) {
                const module = requireFunc(plugin.rendererPath)(appPluginCore);
                await module.install();
            }
        } catch (error) {
            console.error(error);
        }
    })
}
```

### 2.2 插件渲染端入口约定（事实标准）

目前插件渲染端入口文件的约定形态是：

- `module.exports = (appPluginCore) => ({ install: async () => { ... } })`
- 或导出一个对象/函数并提供 `install()` 方法

宿主会执行：`require(rendererPath)(appPluginCore)`，然后 `await module.install()`。

## 四、渲染端“插件核心对象”：`AppRendererPluginCore`

插件拿到的核心对象是 `new AppRendererPluginCore(appRendererCore, pluginName)`，它做了两件事：

1. **把宿主能力以“插件视角”封装一层**（避免插件直接摸到宿主内部结构）
2. **为插件组件注入 mixin**（统一提供插件配置读写、pluginT 等）

## 五、组件包装与 mixin 注入（已实现）

### 5.1 PluginComponent mixin

`PluginComponent` 给每个插件组件注入：

- `computed.$pluginT()`：插件 i18n key 前缀封装
- `setPluginProperty/hasPluginProperty/getPluginProperty/getPluginProperties`：插件配置读写（按 `pluginConfig.${module}.*` 命名空间）

代码：

```11:46:C:\Users\PC\.cursor\worktrees\dubbo-desktop-manager\lye\src\renderer\core\plugin\PluginComponent.js
get() {
    const module = this.#module;
    return {
        computed: {
            $pluginT() {
                return function (key, ...args) {
                    return this.$t(`pluginLocale.${module}.${key}`, ...args);
                };
            }
        },
        methods: {
            setPluginProperty: (key, value) => appConfig.setProperty(`pluginConfig.${module}.${key}`, value),
            hasPluginProperty: (key) => appConfig.hasProperty(`pluginConfig.${module}.${key}`),
            getPluginProperty: (key) => appConfig.getProperty(`pluginConfig.${module}.${key}`),
            getPluginProperties: () => appConfig.getProperties(`pluginConfig.${module}`),
        }
    }
}
```

### 5.2 wrapComponent 做了什么

`wrapComponent()` 会：

- 给组件追加 plugin mixin（避免重复注入）
- 注入一个 `myTabList` 的扩展版（确保插件 tab 中的 component 也会继续 wrap）

这保证了：**插件把组件塞进宿主 tab/dialog/菜单页面后，仍然带着插件能力与命名空间配置。**

## 六、渲染端插件市场/发现机制（现状）

渲染端通过 `PluginProvider` 做 npm registry 搜索、详情获取、推荐安装提示（`ipcRenderer` 监听 `plugin-discovery`）。

它主要负责“发现与展示”，实际安装/卸载仍由主进程 `pluginManager` 执行（渲染端通过 `PluginManagerClient` 调用）。

## 七、当前欠缺点（风险/体验/可维护性）

### 7.1 渲染端 require/代码隔离（高优先级）

当前渲染端也用 `requireFunc(plugin.rendererPath)` 执行插件渲染代码；如果渲染端启用了 nodeIntegration（你工程里是开启的），插件渲染端同样可以拿到 Node 能力。

建议：
- **渲染端也引入类似主进程的沙箱策略**：
  - 基础：给插件的 `appPluginCore` 不暴露危险能力（已做）
  - 进阶：渲染端的 `require`/模块访问也做白名单（或把插件渲染代码放到隔离的 context/iframe/webview）
- **最理想**：关闭渲染端 nodeIntegration + 使用 preload 暴露安全 API（但这是架构级改造）

### 7.2 插件卸载/重载（中高优先级）

目前渲染端加载是一次性 `forEach(async ...)`，没有：
- 插件卸载时的 UI 清理（菜单、设置组件、事件监听、tab、全局组件等）
- 插件重载（更新插件后无需重启）

建议：
- 建立 `PluginRuntime`（每插件一份运行时状态）：
  - 注册的菜单/组件列表
  - event bus listeners
  - 注入的 Vue components 名称
  - 提供 `dispose()` 生命周期，卸载时统一回收

### 7.3 并发加载与错误可观测（中优先级）

现在是 `pluginInfos.forEach(async ...)`，但外部不会 await 这些插件加载完成；加载失败也只是 `console.error`。

建议：
- 改成 `for...of await` 或 `await Promise.allSettled(...)` 并汇总结果
- 将失败信息上报到“插件日志/插件中心 UI”

### 7.4 命名空间与冲突治理（中优先级）

当前 `component(name, component)` 只是 warn 冲突，但仍会注册全局组件。

建议：
- 约定/强制：插件全局组件命名必须带前缀（如 `${pluginName}-xxx`）
- 或只允许“局部组件注入”（通过菜单/页面组件树使用，不允许全局 `Vue.component`）

### 7.5 UI 隔离（CSS/样式污染）（中优先级）

插件 Vue 组件的样式可能污染宿主（全局 CSS）。

建议：
- 约定插件组件尽量使用 scoped CSS
- 或在宿主容器上加命名空间 class，配合插件样式前缀
- 更强隔离：webview/iframe（成本更高）

### 7.6 主进程 API 调用鉴权（中优先级）

渲染端通过 `ApiConsumer` 调用主进程模块，目前主进程侧没有可靠的“调用者身份”。

建议：
- 在 IPC/HTTP invocation 中加入 `callerPluginName`（由渲染端 `AppRendererPluginCore.referencePluginService` 自动附加）
- 主进程 `ApiExportor` 按 caller 做权限校验（与主进程沙箱权限系统打通）

## 八、建议补齐/增强的“渲染端插件能力”清单（可扩展）

下面是建议新增/增强的能力（按价值排序）：

### 8.1 生命周期与资源回收（必需）

- **`onDispose(fn)`**：插件注册销毁回调
- **`dispose()`**：宿主卸载插件时调用，回收：
  - event bus listener
  - DOM/dialog 实例
  - 动态注册的组件
  - 定时器/interval（可在 core 中包一层 `setTimeout`/`setInterval` 方便追踪）

### 8.2 渲染端日志与调试面板（强烈建议）

- **`logger`**：插件 logger（带插件名标签），输出到插件日志页
- **`reportError(err)`**：统一错误上报（含 stack、版本、环境）

### 8.3 统一的“路由/页面注册”（建议）

目前更多是“菜单 + component”；可以进一步抽象成：
- `registerRoute({ path, component, meta })`
- `registerSidebarMenu({ ... })`（内部生成 route）

### 8.4 统一的数据存储（建议）

当前是 `appConfig` 的 key-value；可以补：
- **`pluginStorage`**：类似 localForage/IndexedDB 的命名空间存储（配额、可清理）
- **数据迁移钩子**：插件升级时迁移配置结构

### 8.5 事件总线命名空间（建议）

避免插件之间事件名冲突：
- `emit('xxx')` 实际发 `plugin:${pluginName}:xxx`
- 提供 `emitGlobal`（明确跨插件）

### 8.6 UI 能力扩展（按需）

- Toast/Notification 的统一封装（带插件名）
- 状态栏/工具栏按钮扩展点
- 右键菜单扩展（与你 `addPluginMenu/fillPluginMenu` 类似但作用于 UI 右键）
